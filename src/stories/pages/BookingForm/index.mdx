## Overview

This demo implements a simple booking form with the following fields:

- **Name** (required)
- **Email** (required, must be a valid email format)
- **Start Date & Time** (required)
- **End Date & Time** (required, must occur after Start Date/Time)
- **Remote Meeting?** (optional checkbox)
- **Comments** (optional textarea)

### Validation Rules

- Name, Email, Start Date/Time, End Date/Time are required.
- Email must be a valid format.
- End Date/Time must be later than Start Date/Time.
- Remote Meeting? and Comments are optional.

## Purpose

The goal of this demo is to illustrate how to handle cross-field validation in forms. While individual field rules (e.g. "email must be valid") are straightforward, checking relationships between multiple fields—such as ensuring the end of a booking is after the start—requires more advanced validation logic.

This scenario is deliberately chosen because HTML5 validation alone struggles here. Native browser validation can flag missing values or malformed emails but it cannot easily express rules like `End time must be after Start time.` Even if you manage to wire it up, the resulting UX is inconsistent and hard to manage.

## Key Points

- **Conditional validation** - The `End Date` and `End Time` fields are validated against the chosen `Start Date` and `Start Time`. If the end occurs before the start, a custom error message is displayed.
- **Improved UX** - Instead of relying on native validation popups (which vary by browser and cannot always be styled consistently), the demo shows how you can display validation errors inline, next to the relevant field(s), for a smoother experience.
- **Error state handling** - In comparison to the [React Hook Form alternative](../?path=/docs/reacthookform-bookingform--docs) this example highlights how RHF makes it easier to manage complex rules and maintain a consistent error display across the form.
- **Maintainability** - The handleChange and handleBlur functions grow more complex when custom rules are layered on top of HTML5's built-ins.

### Pros

- **Zero setup** – no extra libraries, schema, or framework required.
- **Good for** small/simple forms – native attributes cover most basic rules.
- **Performance –** no schema parsing or extra validation layer; browser handles it.

### Cons

- **Limited flexibility** – can't easily enforce cross-field rules like "end after start" without custom JavaScript.
- **Inconsistent UX** – native popups look different across browsers and can't be styled.
- **Scaling issues** – as forms grow, juggling HTML5 attributes with custom JS logic leads to messy event handlers.
- **Accessibility quirks** – native tooltips aren’t always screen-reader friendly, and inline error summaries must be built manually.
