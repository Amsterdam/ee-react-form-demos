## Overview

This demo implements the NPM package [@amsterdam/ee-ads-rhf](https://www.npmjs.com/package/@amsterdam/ee-ads-rhf) variant of the booking form with the following fields:

- **Name** (required)
- **Email** (required, must be a valid email format)
- **Start Date & Time** (required)
- **End Date & Time** (required, must occur after Start Date/Time)
- **Comments** (optional textarea)

### Validation Rules

- Name, Email, Start Date/Time, End Date/Time are required.
- Email must be a valid format.
- End Date/Time must be later than Start Date/Time.
- The Comments field is optional.

## Purpose

This variant demonstrates the same RHF booking flow as the standard examples, but implemented through `@amsterdam/ee-ads-rhf` abstractions.

In this case, the form must ensure that the end date/time occurs after the start date/time. HTML5 alone cannot reliably enforce this, but with RHF (and optionally a [Zod](https://zod.dev/) schema), the rule is expressed cleanly and enforced consistently.

## Key Points

- **Package variant** - Behavior is intentionally aligned with the RHF Booking Form demo, while form integration is routed through `@amsterdam/ee-ads-rhf`.
- **Cross-field validation** - RHF integrates with schema validation or custom rules, making it easy to compare related fields (e.g. end vs start).
- **Consistent UX** - Instead of browser tooltips, errors are displayed inline, styled with the design system.
- **Error state management** - RHF automatically tracks touched/dirty states and surfaces errors in a predictable way.
- **Cleaner developer experience** - With `zodResolver` or RHF’s built-in rules, validation logic is centralized and easier to maintain.

### Pros

- **Advanced validation support** – cross-field rules are expressed clearly and enforced consistently.
- **Better UX** – inline errors are consistent, accessible and not tied to browser quirks.
- **Scalable** – adding more conditional fields or rules is straightforward.
- **State handling built in** – RHF tracks touched, dirty and error states without boilerplate.

### Cons

- **Extra setup** – requires RHF setup and optional schema integration compared to HTML5 attributes.
- **Learning curve** – developers need to understand RHF concepts (e.g. `register`, `watch`, `resolver`).
- **Overhead for small forms** – for a very simple form, RHF may feel heavier than necessary.

When consuming this package variant, an additional consideration is package versioning and release synchronization across consuming apps.
