## Overview

This demo shows the NPM package [@amsterdam/ee-ads-rhf](https://www.npmjs.com/package/@amsterdam/ee-ads-rhf) variant of the contact form with three required fields:

- **Name**
- **Email** (must be valid format)
- **Message**

## Purpose

This Contact Form variant demonstrates the same basic validation strategy as the standard demo (all fields validate on submit), but implemented with the internal `@amsterdam/ee-ads-rhf` package.

Use this page when you want the equivalent demo behavior while integrating through the EE ADS RHF abstractions.

### Validation Rules

- All fields must be filled out.
- The email must be in a valid format.
- Error messages are only displayed after the user clicks Submit.

## Key Points

- **Package variant** - Behavior is intentionally aligned with the non-package Contact Form demo, while field wiring is handled via `@amsterdam/ee-ads-rhf`.
- **Batch feedback** - All validation errors are shown at once after the submit attempt.
- **Simple implementation** - This approach is straightforward to wire up.

### Pros

- **Lightweight & fast to build** – minimal boilerplate and validation logic required.
- **Great fit for short forms** – works well when there are only a few required fields.
- **Less cognitive load** – users aren’t distracted by real-time errors while typing.
- **Predictable UX** – users know exactly when validation happens (only on submit).
- **Easier maintenance** – fewer moving parts compared to `onChange` validation.

### Cons

- **Risk of frustration** - If multiple fields are incorrect, the user only finds out after completing the entire form.
- **Delayed feedback** - Typos or mistakes aren’t caught until the end, which can increase the number of corrections needed.
- **Doesn’t scale well** – as forms get longer or rules more complex, this approach becomes clunky.
