## Overview

This demo implements a complex “Create Entity” form that collects data for generating a [Backstage](https://backstage.io/) entity YAML file.

Fields

- **Kind** – select menu (required)
- **Name** – input (required)
- **Description** – textarea
- **Type** – select menu (required)
- **Lifecycle** – select menu (required)
- **Owner** – react-select (single value, required)
- **Has System?** – checkbox
- **System** – `react-select` (required only if `Has System` is checked)
- **Tags** – `react-select` (multiple values)
- **Annotation Repeater** – dynamic rows:
  - Key – `react-select` (required)
  - Value – input/select (required)
- **Links Repeater** – dynamic rows:
  - URL – input (required)
  - Title – input (required)
  - Icon – select (required)

## Purpose

This variant builds on the [HTML5 Zod validation example](../?path=/docs/plain-html5-createentity-zod-validation--docs) but integrates it directly with [React Hook Form](https://react-hook-form.com/) using `zodResolver`.

The goal is to show how schema-based validation can be combined with a modern form library to:

- Reduce boilerplate (state, refs, error handling are managed automatically).
- Improve performance (fewer unnecessary re-renders).
- Scale easily with complex, dynamic forms.
- Provide a unified, ergonomic developer experience.

### Validation Rules

- Required fields (Kind, Name, Type, Lifecycle, Owner).
- System required if Has System is checked.
- Annotation rows: Key + Value required.
- Link rows: URL, Title, Icon required.

## Key Points

- **RHF + Zod Integration** - Using the `zodResolver`, the schema drives validation while RHF handles form state.
- **Cleaner state management** - No need to manually track touched/dirty states or error objects — RHF handles it.
- **Dynamic arrays made easier** - Repeater fields use RHF’s `useFieldArray`, which integrates smoothly with Zod array schemas.
- **Better performance** - RHF is designed to minimize re-renders, even in large/dynamic forms.

### Pros

- **Developer-friendly** – validation, touched state and error handling are all unified under one system.
- **Scalable** – easily extends to new fields and repeaters with minimal boilerplate.
- **Consistent UX** – Zod ensures rules are declarative, RHF ensures errors are shown consistently.
- **Optimized rendering** – large/dynamic forms stay responsive.

### Cons

- **Learning curve** – requires understanding both RHF’s API and Zod schemas.
- **Setup cost** – slightly more code/config compared to plain HTML5 forms.
