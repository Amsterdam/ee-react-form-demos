## Overview

This demo implements a simple booking form with the following fields:

- **Name** (required)
- **Email** (required, must be a valid email format)
- **Start Date & Time** (required)
- **End Date & Time** (required, must occur after Start Date/Time)
- **Comments** (optional textarea)

### Validation Rules

- Name, Email, Start Date/Time, End Date/Time are required.
- Email must be a valid format.
- End Date/Time must be later than Start Date/Time.
- The Comments field is optional.

## Purpose

The goal of this demo is to illustrate how to handle reusable cross-field validation in [React Hook Form](https://react-hook-form.com/) based forms. In this example we replace HTML5's built-in validation with a [Zod](https://zod.dev/) schema for a consistent validation strategy.

With [Zod](https://zod.dev/) you can centralize all rules, including cross-field checks like "end must be after start", in one schema and keep them framework-agnostic. This approach improves maintainability, readability and scalability compared to either native HTML5 validation or RHF alone.

This form validates on submit. Inline errors and the main error alert only appear after a submit attempt, keeping the interface uncluttered during typing while still providing clear feedback when validation fails.

## Key Points

- **Schema-driven validation** - All rules are defined in one [Zod](https://zod.dev/) schema, keeping form logic clear and maintainable.
- **Cross-field refinement** - The schema enforces that End Date/Time is after Start Date/Time, with a single, centralized rule.
- **Tight integration** - Using RHF’s `zodResolver`, validation is automatic and errors flow naturally into RHF’s error state.
- **Better dev experience** - By separating validation from rendering logic, the form is easier to read and extend.

### Pros

- **Centralized validation** – rules live in one place (the schema), reducing duplication and hidden logic.
- **Expressive** – [Zod](https://zod.dev/) allows complex, cross-field checks in a declarative style.
- **Predictable UX** – errors are surfaced inline and consistently styled, avoiding browser inconsistencies.
- **Scalable** – easily extended to handle more fields, relationships, or conditional logic.

### Cons

- **Extra dependency** – adds [Zod](https://zod.dev/) as well as RHF.
- **Schema complexity** – for developers unfamiliar with [Zod](https://zod.dev/), `.refine()` and custom error shaping may be a hurdle.
- **Overhead for tiny forms** – might feel heavier than needed for simple two-field forms.
