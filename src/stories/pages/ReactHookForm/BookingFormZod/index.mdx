## Overview

This demo implements a simple booking form with the following fields:

- **Name** (required)
- **Email** (required, must be a valid email format)
- **Start Date & Time** (required)
- **End Date & Time** (required, must occur after Start Date/Time)
- **Remote Meeting?** (optional checkbox)
- **Comments** (optional textarea)

### Validation Rules

- Name, Email, Start Date/Time, End Date/Time are required.
- Email must be a valid format.
- End Date/Time must be later than Start Date/Time.
- Remote Meeting? and Comments are optional.

## Purpose

This variant shows how [React Hook Form](https://react-hook-form.com/) and [Zod](https://zod.dev/) can work together to handle both simple and complex form validation.

The Zod schema centralizes all rules, including the cross-field constraint that End Date/Time must occur after Start Date/Time. This approach improves maintainability, readability, and scalability compared to either native HTML5 validation or RHF alone.

## Key Points

- **Schema-driven validation**: All rules are defined in one Zod schema, keeping form logic clear and maintainable.
- **Cross-field refinement**: The schema enforces that End Date/Time is after Start Date/Time, with a single, centralized rule.
- **Tight integration**: Using RHF’s zodResolver, validation is automatic and errors flow naturally into RHF’s error state.
- **Better dev experience**: By separating validation from rendering logic, the form is easier to reason about and extend.

- **Pro**: **Centralized validation** – rules live in one place (the schema), reducing duplication and hidden logic.
- **Pro**: **Expressive** – Zod allows complex, cross-field checks in a declarative style.
- **Pro**: **Predictable UX** – errors are surfaced inline and consistently styled, avoiding browser inconsistencies.
- **Pro**: **Scalable** – easily extended to handle more fields, relationships, or conditional logic.
- **Con**: Extra dependency – adds Zod as well as RHF.
- **Con**: **Schema complexity** – for developers unfamiliar with Zod, .refine() and custom error shaping may be a hurdle.
- **Con**: **Overhead for tiny forms** – might feel heavier than needed for simple two-field forms.
