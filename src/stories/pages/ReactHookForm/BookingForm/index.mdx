## Overview

This demo implements a simple booking form with the following fields:

- **Name** (required)
- **Email** (required, must be a valid email format)
- **Start Date & Time** (required)
- **End Date & Time** (required, must occur after Start Date/Time)
- **Remote Meeting?** (optional checkbox)
- **Comments** (optional textarea)

### Validation Rules

- Name, Email, Start Date/Time, End Date/Time are required.
- Email must be a valid format.
- End Date/Time must be later than Start Date/Time.
- Remote Meeting? and Comments are optional.

## Purpose

This variant demonstrates how [React Hook Form](https://react-hook-form.com/) makes cross-field validation more straightforward and provides a smoother UX than native HTML5 validation.

In this case, the form must ensure that the end date/time occurs after the start date/time. HTML5 alone cannot reliably enforce this, but with RHF (and optionally a Zod schema), the rule is expressed cleanly and enforced consistently.

## Key Points

- **Cross-field validation**: RHF integrates with schema validation or custom rules, making it easy to compare related fields (e.g. end vs start).
- **Consistent UX**: Instead of browser tooltips, errors are displayed inline, styled with the design system.
- **Error state management**: RHF automatically tracks touched/dirty states and surfaces errors in a predictable way.
- **Cleaner developer experience**: With `zodResolver` or RHF’s built-in rules, validation logic is centralized and easier to maintain.

- **Pro**: **Advanced validation support** – cross-field rules are expressed clearly and enforced consistently.
- **Pro**: **Better UX** – inline errors are consistent, accessible and not tied to browser quirks.
- **Pro**: **Scalable** – adding more conditional fields or rules is straightforward.
- **Pro**: **State handling built in** – RHF tracks touched, dirty and error states without boilerplate.
- **Con**: **Extra setup** – requires RHF setup and optional schema integration compared to HTML5 attributes.
- **Con**: **Learning curve** – developers need to understand RHF concepts (e.g. register, watch, resolver).
- **Con**: **Overhead for small forms** – for a very simple form, RHF may feel heavier than necessary.
