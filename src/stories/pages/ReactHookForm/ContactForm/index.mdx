## Overview

This demo extends the same [HTML 5 contact form example](../?path=/docs/reacthookform-contactform--docs), this time built with React Hook Form and a Zod schema for validation.

### Validation Rules

- All fields must be filled out.
- The email must be in a valid format.
- Error messages are only displayed after the user clicks Submit.

- **Name**
- **Email** (must be valid format)
- **Message**

## Purpose

This variant demonstrates how [React Hook Form (RHF)](https://react-hook-form.com/) and [Zod](https://zod.dev/) can work together to provide the same "validate on submit" (or alternatively, on change, via the `mode` option property) UX strategy as the HTML5 version but with a cleaner and more scalable developer experience.

Instead of relying on the browser’s built-in validation, the form uses a [Zod](https://zod.dev/) schema to define rules and RHF’s `zodResolver` to enforce them. This ensures consistent error messages, predictable validation flow and easier extension if the form grows.

## Key Points

- **Schema-driven**: Validation rules are centralized in a Zod schema, avoiding browser-specific quirks.
- **RHF integration**: The zodResolver bridges Zod and React Hook Form, so errors are handled automatically.
- **Same UX strategy**: Errors only appear on submit, keeping the typing experience distraction-free.
- **Consistent messaging**: All validation errors follow the same format and styling, regardless of browser.

- **Pro**: **Cleaner developer experience** – no need to wire up custom state or rely on browser tooltips.
- **Pro**: **Easier to extend** – if more fields are added later, they can be slotted into the schema without changing form logic.
- **Pro**: **Consistency** – using the same Zod schema for both frontend and backend is possible.
- **Con**: **Extra setup** – requires RHF + Zod setup compared to the simplicity of required attributes.
- **Con**: **Overhead for tiny forms** – might feel heavier than necessary for a very small form like this.
