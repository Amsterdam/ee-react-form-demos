# React 19 - Server Components

> **Warning**
> This project is a work in progress so will likely change in the future.

- React Server Components (RSC) / Next.js
  - In React v19, `useActionState` was introduced.
  - `useActionState` is a React hook that helps to manage the state of a form submission handled by an action function (for example, a server action). It helps manage things like:
    - The form submitting state (for example: `isPending`, `isSubmitting`)
    - The server's response/result
  - Next.js popularized it by using it with server actions and form handling.
  - In Next.js, forms can call server functions directly, for example:
    `async function myServerAction(prevState, formData) { ... }`
    `<form onSubmit={myServerAction}>...</form>`
    This code works seamlessly with `useActionState`
  - However, to run Server Components, the framework needs to know how to serialize and transport `myServerAction` to the server, which requires a special compiler and runtime support (React Server actions).
  - Therefore, implementing these requires a bundler setup that supports Server Actions and React Server Components. This includes bundlers like Vite (still considered experimental) or framework like Next.js.
    - This could also complicate your devops setup, if you are currently deploying your React based frontend as a static web application.
    - If you also already have separate backend application that serves as an API, this makes the backend-frontend separation less clear. Introducing Server Actions is effectively a full-stack architecture change.

Tldr:

- React 19 introduced powerful new tools like `useActionState` and Server Actions, which effectively enable backend-like functionality on the frontend. Howver, this depends on bundler and runtime support, which is currently only provided by Next.js App Router.
  - Remix/react-router + Vite can do SSR but not the new RSC-based Server Actions model yet.
- Adopting these features means the frontend is no longer 'frontend-only' - it requires Node-compatible runtimes and backend hosting environments.
- Therefore, for now, Server Actions are an opt-in forward-looking feature - not a requirement to adopt across all projects.
