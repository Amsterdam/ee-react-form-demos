# Testing - Overview

Forms are one of the highest-risk parts of a web application.

They combine user interaction, validation rules, async submission, API integration, accessibility and edge cases — all in one place.

This guide explains what to test and common pitfalls when testing forms in React.

## TL;DR

When testing a form, make sure you have tests for:

- Users can complete and submit the form successfully (happy path)
- Required and invalid inputs show clear validation errors
- Server-side errors are displayed correctly
- Submission behaviour is correct (loading state, prevents double submit)
- Conditional logic works (dynamic fields, repeaters, multi-step flows)
- Accessibility and keyboard navigation are not accidentally broken
- The correct payload is sent to the backend

---

## Minimum recommended tests per form

Every form should *at least* have tests for:

### Level 1 (minimum baseline)

- Renders without crashing
- Happy path submit works
- Required validation works

### Level 2 (recommended for most forms)

- Invalid format validation (email/date/postcode etc.)
- Server-side errors display correctly
- Submit button disables during loading
- Error recovery works (fail → fix → submit success)

### Level 3 (critical / complex forms)

- Conditional fields
- Repeaters (add/remove)
- Multi-step wizard navigation
- File uploads
- Accessibility checks + keyboard navigation

---

## Recommended testing strategy

Use a combination of:

- **Unit / integration tests** (fast feedback, local, most coverage)
- **End-to-end tests** (realistic browser flow, slower but high confidence)

### Tools

- **Unit/integration tests**: Vitest/Jest + React Testing Library
- **E2E tests**: Cypress or Playwright
- **Accessibility checks**: `@axe-core/playwright` or `cypress-axe`

---

## What to test (checklist)

### Must-have tests (high value)

- **Happy path submission**
  - Fill in required fields
  - Submit
  - Assert success UI or navigation occurs

- **Required field validation**
  - Submit with empty required fields
  - Assert errors are visible

- **Invalid format validation**
  - Email, date, phone, number ranges, etc.
  - Assert correct message appears

- **Server-side error handling**
  - Mock API returning validation errors
  - Assert correct error message appears near field and in summary

- **Loading state**
  - Submit button disables while request is in progress
  - Prevents double submission

- **Conditional fields**
  - Selecting option A shows extra fields
  - Option B hides them again
  - Hidden fields should not submit invalid values

- **Repeaters / dynamic collections**
  - Add/remove row
  - Ensure correct row updates
  - Ensure submission payload includes correct list values

---

### Medium priority tests (important depending on complexity)

- **Error recovery**
  - First submit fails
  - Second submit succeeds after correction

- **Default values**
  - Existing values render correctly (edit forms)
  - Form reset behaviour works

- **Multi-step flows**
  - Step transitions work
  - Validation blocks next step
  - Data persists across steps

- **Keyboard navigation**
  - `Tab` order makes sense
  - `Enter` triggers submission correctly

---

## Examples

### Example from `ContactFormLive.test.tsx`:

```tsx
it('should update form data when user types', async () => {
  const user = userEvent.setup();
  render(<ContactFormLive />);

  const nameInput = screen.getByLabelText('Naam');
  const emailInput = screen.getByLabelText('E-mailadres');
  const bodyInput = screen.getByLabelText('Bericht');

  await user.type(nameInput, 'John Doe');
  await user.type(emailInput, 'john@example.com');
  await user.type(bodyInput, 'Test message');

  expect(nameInput).toHaveValue('John Doe');
  expect(emailInput).toHaveValue('john@example.com');
  expect(bodyInput).toHaveValue('Test message');
});
```

### End-to-End Testing with Cypress/Playwright

Test forms in a real browser environment, catching issues unit tests might miss — such as integration with the backend, CSS and actual navigation.

#### What E2E Covers

- Full form flow (fill, submit, redirect)
- Network requests and API integration
- Accessibility and keyboard navigation
- File uploads in a real environment
- Edge cases with actual browser behavior

#### Example using [Cypress](https://www.cypress.io):

```tsx
it('completes booking form and submits', () => {
  cy.visit('/booking');
  cy.findByLabelText(/Name/i).type('Jane Smith');
  cy.findByLabelText(/Email/i).type('jane@example.com');
  cy.findByLabelText(/Start Date/i).type('2025-08-20');
  cy.findByLabelText(/Start Time/i).type('10:00');
  cy.findByRole('button', { name: /Submit/i }).click();

  cy.contains(/Booking confirmed/i).should('be.visible');
});
```

#### Example using [Playwright](https://playwright.dev):

```tsx
test('shows validation error when email is missing', async ({ page }) => {
  await page.goto('/contact');
  await page.click('button[type=submit]');
  await expect(page.getByText(/email is required/i)).toBeVisible();
});
```
