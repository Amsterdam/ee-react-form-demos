# How to Handle Validation

> **Warning**
> Frontend validation is only for UX, not security. All validation must also be enforced on the backend. Frontend forms can still guide users toward valid input and prevent common mistakes.

This document explains how to handle form validation errors and is based on the [GOV.UK Design System Validation Patterns document](https://design-system.service.gov.uk/patterns/validation/)..

Validation errors happen when a user enters data that is missing, incorrect, ambiguous, or unusable. The goal is to communicate these errors clearly and accessibly so users can fix them and continue.

Follow this pattern whenever you need to check user-provided information and provide feedback when something is invalid or missing.

## 1. Validate inputs appropriately

- Reject information that is clearly incorrect, too ambiguous or missing (when required).
- Accept reasonable variations in format (for example, a postcode with or without a space).
- Strip out or ignore unnecessary characters such as extra spaces or punctuation where possible.

## 2. Show validation errors clearly

When validation fails:

- Re-render the page with the form fields preserved as the user entered them.
- Show an [Invalid Form Alert](https://designsystem.amsterdam/?path=/docs/components-forms-invalid-form-alert--docs) at the top of the main container and move keyboard focus to it.
- Add `(X invoerfouten)` to the start of the page `<title>` so screen readers announce the error state immediately (the Invalid Form Alert does this automatically).
- Show [Error message](https://designsystem.amsterdam/?path=/docs/components-forms-error-message--docs) components next to each field with an error.

See also: [Amsterdam Design System - Form Flow](https://designsystem.amsterdam/?path=/docs/pages-public-form-flow--docs)

## 3. When to validate

- Validate when the user attempts to continue (for example after clicking Continue/Submit).
- Avoid validating on change (while the user is still typing), unless there is evidence it improves usability.

## 4. Client vs. server validation

- Always implement server-side validation (it is the only reliable enforcement).
- Only add client-side validation if there is a clear user benefit.

## 5. Turn off built-in HTML5 validation

Default browser validation cannot be styled or made accessible in a way consistent with [Amsterdam Design System](https://designsystem.amsterdam/) standards:

- Add `novalidate` to the `<form>` tag.
- Don't use `required` attributes on inputs.
