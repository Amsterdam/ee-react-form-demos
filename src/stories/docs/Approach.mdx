# Which Approach Should I Use?

The following guide should assist you in deciding on a solution to build your form.

1. What scales well for tiny vs. complex forms
2. Tradeoffs: simplicity vs. flexibility vs. UX control
3. When to use HTML5 + [Zod](https://zod.dev/), or [React Hook Form (RHF)](https://react-hook-form.com/) + [Zod](https://zod.dev/)

## Choosing a Form Validation Strategy

Different forms call for different levels of validation sophistication. The demos in this project showcase three main approaches:

- **HTML5 + Zod schema validation** - middle ground, more control
- **[React Hook Form (RHF)](https://react-hook-form.com/) + [Zod](https://zod.dev/) schema validation** - most powerful, best for scaling

## Overview

| Form Size / Complexity                                       | Recommended Approach             | Why                                                                                 |
| ------------------------------------------------------------ | -------------------------------- | ----------------------------------------------------------------------------------- |
| **Small, simple form** (e.g. Contact form)                   | **HTML5 + Zod**                  | Better UX than HTML5-only, but still lightweight.                                   |
| **Medium form with cross-field rules** (e.g. Booking form)   | **HTML5 + Zod** or **RHF + Zod** | Zod handles logic; RHF adds smoother state handling if more fields are added later. |
| **Large / dynamic form** (e.g. Create Entity with repeaters) | **RHF + Zod**                    | Best scalability, optimized rendering, declarative validation.                      |


Below is an overview of their strengths, weaknesses and recommended use cases.

### 1. HTML5 + Zod Schema Validation

**Best for:** small to medium forms where you need custom rules but don't want the overhead of a form library.

#### Pros

- Centralized validation logic (schemas can be reused server-side).
- Supports advanced rules HTML5 can't (e.g. `end date > start date`)
- Consistent, styled inline errors instead of native popups.
- Balanced UX: inline errors appear while typing but global error alerts only show after submit.

#### Cons

- More boilerplate than HTML5.
- Must manage error state manually (`errors`, `submitTouched`).
- No built-in touched/dirty tracking.

### 2. React Hook Form (RHF) + Zod Schema

**Beast for:** medium to large forms, epsecially with repeaters, conditional fields or cross-field validation rules.

#### Pros

- Full state management built-in (touched, dirty, error states).
- Scales smoothly to large/dynamic forms with minimal boilerplate.
- Optimized re-renders keep big forms performant.
- Declarative, reusable validation (Zod schema).
- Consistent and accessible error display across all fields.

#### Cons

- Setup cost + learning curve (RHF concepts, resolvers).
- Overkill for very small forms.
